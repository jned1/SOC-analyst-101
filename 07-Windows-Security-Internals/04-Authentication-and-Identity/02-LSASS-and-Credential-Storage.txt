# 02 - LSASS and Credential Storage

## Overview

The Local Security Authority Subsystem Service (LSASS) is one of the most critical security processes in Windows. It is responsible for enforcing the system's security policy and handling authentication requests for users and services.

LSASS manages credential validation, generates security tokens for authenticated users, and coordinates communication with authentication mechanisms such as NTLM and Kerberos. Because it processes and stores sensitive authentication material, LSASS is considered one of the most sensitive processes in the operating system.

Any compromise of LSASS can expose credentials, authentication tokens, and session data, allowing attackers to impersonate users, escalate privileges, or move laterally across a network.

Understanding how LSASS operates is essential for both defenders and attackers. From a defensive perspective, monitoring LSASS activity provides critical visibility into authentication behavior and credential-related attacks.

---

## Role of LSASS in Windows Security

LSASS acts as the core authority for authentication and security policy enforcement within Windows. It operates as the central component responsible for validating identities and establishing trusted sessions.

### User Authentication

When a user attempts to log in, the Windows authentication subsystem forwards the credentials to LSASS. LSASS then interacts with the appropriate authentication package to verify the identity.

Depending on the authentication method, verification may occur against:

- Local account databases
- Domain controllers
- Authentication services using Kerberos or NTLM

Successful authentication results in the creation of a user security context.

---

### Access Token Generation

After verifying credentials, LSASS generates an access token representing the authenticated identity. This token becomes the security identity used by the operating system when executing processes on behalf of the user.

The access token contains:

- User Security Identifier (SID)
- Group memberships
- Privileges assigned to the account
- Authentication information

All processes launched within the user session inherit this access token.

---

### Password Verification

For local accounts, LSASS validates credentials by comparing password hashes stored in the Security Accounts Manager database.

For domain accounts, authentication typically involves communication with a domain controller using Kerberos or NTLM challenge-response mechanisms.

In both cases, LSASS ensures that the provided credentials match the stored authentication data.

---

### Security Policy Enforcement

LSASS enforces local security policies defined on the system. These policies control behaviors such as:

- Password policies
- Account lockout rules
- Logon restrictions
- Audit settings

Security policy enforcement ensures consistent authentication rules across the operating system.

---

### Credential Caching

Windows systems may cache certain authentication data to support offline logon scenarios or improve authentication performance.

For example:

- Cached domain credentials allow users to log in even when a domain controller is unavailable.
- Kerberos tickets may be stored in memory for reuse during service authentication.

This cached information often resides in LSASS memory.

---

## Authentication Packages Managed by LSASS

LSASS relies on authentication packages to implement specific authentication protocols. These packages process credential information and perform the actual identity validation.

### NTLM

NTLM is a legacy challenge-response authentication protocol commonly used for compatibility with older systems or in environments where Kerberos is unavailable.

NTLM authentication works by verifying password hashes without transmitting plaintext credentials across the network.

Simplified NTLM process:

    Client sends authentication request
        -> Server issues challenge
            -> Client returns hashed response
                -> Server verifies hash

Although widely supported, NTLM is considered less secure than Kerberos.

---

### Kerberos

Kerberos is the primary authentication protocol used in Active Directory environments.

Kerberos relies on a ticket-based system that allows secure authentication without repeatedly transmitting credentials.

Typical Kerberos flow:

    User authenticates
        -> Domain controller issues Ticket Granting Ticket (TGT)
            -> Client requests service ticket
                -> Ticket used to access services

Kerberos tickets may be stored in memory within LSASS during an active session.

---

### Negotiate (SPNEGO)

The Negotiate protocol, also known as SPNEGO, determines which authentication protocol should be used between NTLM and Kerberos.

When authentication begins, Negotiate evaluates the environment and selects the most appropriate mechanism.

Example decision process:

    Domain environment available
        -> Use Kerberos
    Otherwise
        -> Fallback to NTLM

This negotiation ensures compatibility across different environments.

---

## Credential Storage Mechanisms

Credentials and authentication artifacts may exist in several locations within a Windows system.

### LSASS Memory Storage

LSASS stores various authentication-related data in memory during active sessions. This may include:

- Credential material used for authentication
- Kerberos tickets
- Session tokens
- Cached credentials

Because these artifacts reside in memory, attackers often attempt to access LSASS memory to extract them.

---

### Cached Domain Credentials

When domain users log in, Windows may store cached credential data locally to support offline authentication.

These cached credentials allow users to log in even if the system cannot contact the domain controller.

Cached credentials are stored securely but can still be targeted by attackers with sufficient system privileges.

---

### Kerberos Tickets

Kerberos authentication generates tickets that allow users to access network services without repeatedly entering credentials.

These tickets may exist in LSASS memory during an authenticated session.

Attackers may attempt to extract these tickets for reuse in authentication attacks.

---

### Security Accounts Manager (SAM) Database

The Security Accounts Manager database stores local user account information and password hashes.

LSASS interacts with the SAM database when validating credentials for local accounts.

Access to the SAM database is highly restricted because it contains sensitive authentication data.

---

### Credential Manager

Windows Credential Manager allows applications and services to store authentication credentials securely.

Stored credentials may include:

- Network authentication credentials
- Application login information
- Remote system credentials

Although protected, these credentials may become targets if attackers gain sufficient system access.

---

## Access Tokens and Identity Representation

After successful authentication, LSASS generates an access token that represents the user's identity.

### Token Creation Process

The token creation process includes:

    Credentials verified
        -> User SID identified
            -> Group memberships retrieved
                -> Privileges assigned
                    -> Access token generated

This token becomes the identity used by the system when evaluating access control decisions.

---

### Security Identifiers (SIDs)

A Security Identifier uniquely represents a user or group within Windows.

Access tokens include the user's SID as well as SIDs associated with groups to which the user belongs.

These identifiers are used by the operating system to evaluate access control lists.

---

### Privileges and Group Memberships

Access tokens also include privileges assigned to the user account.

Examples of privileges include:

- Administrative rights
- System management privileges
- Resource access permissions

These privileges determine what actions the user can perform within the system.

---

## Security Risks and Attack Techniques

Because LSASS manages authentication secrets, it is a primary target for attackers attempting to steal credentials or impersonate users.

### Credential Dumping

Credential dumping involves extracting authentication data from memory or protected storage locations.

Attackers may target LSASS memory to retrieve:

- Password hashes
- Kerberos tickets
- Authentication tokens

This data can then be used for further attacks.

---

### LSASS Memory Extraction

Attackers may attempt to read the memory of the LSASS process directly.

If successful, they can obtain authentication material stored during active user sessions.

This technique often requires elevated privileges on the system.

---

### Pass-the-Hash Attacks

Pass-the-Hash attacks reuse stolen NTLM password hashes to authenticate to other systems.

The attacker does not need to know the original password. Instead, the hash is injected into an authentication request.

---

### Pass-the-Ticket Attacks

Pass-the-Ticket attacks involve stealing Kerberos tickets from memory and reusing them to access network resources.

Because the ticket represents authenticated identity, possession of the ticket can allow unauthorized access.

---

### Abuse of Authentication Tokens

Attackers may attempt to impersonate users by manipulating authentication tokens.

If an attacker obtains a token belonging to a privileged user, they may gain elevated access to system resources.

---

## Defensive and SOC Perspective

Defenders monitor LSASS activity closely because it is closely tied to authentication behavior and credential handling.

### Monitoring Access to LSASS Memory

Unauthorized attempts to access LSASS memory are often strong indicators of credential theft activity.

Security tools monitor processes attempting to read or interact with LSASS.

---

### Suspicious Process Behavior Targeting LSASS

Processes that attempt to open handles to LSASS or perform memory operations on it may indicate malicious activity.

Examples include tools attempting to dump credentials from memory.

---

### Authentication Event Monitoring

Windows event logs record authentication activity and may reveal abnormal patterns such as:

- Repeated authentication failures
- Unusual logon types
- Authentication attempts from unexpected systems

Such anomalies may indicate credential abuse.

---

### Endpoint Protection Mechanisms

Modern endpoint detection systems monitor behavioral patterns associated with credential theft attempts.

Examples include:

- Processes accessing LSASS memory
- Suspicious authentication activity
- Credential dumping behavior

These alerts help analysts identify potential identity-based attacks.

---

## Mitigations and Protection Mechanisms

Windows includes several mechanisms designed to protect LSASS and credential data.

### LSASS Protection (RunAsPPL)

LSASS can run as a Protected Process Light (PPL), which restricts access from non-protected processes.

This protection prevents many tools from directly accessing LSASS memory.

---

### Credential Guard

Credential Guard isolates sensitive authentication secrets using virtualization-based security.

Credentials are stored in a protected environment that is inaccessible to the normal operating system.

---

### Protected Process Light (PPL)

Protected Process Light restricts which processes can interact with protected system services such as LSASS.

Only trusted and signed processes are allowed to interact with protected components.

---

### Least Privilege Enforcement

Limiting administrative privileges reduces the likelihood that attackers can access LSASS memory or perform credential dumping.

Proper privilege management is a fundamental defensive strategy.

---

## Key Takeaways

LSASS is the central component responsible for authentication enforcement and identity management in Windows.

It validates credentials, generates access tokens, manages authentication packages, and stores sensitive authentication data during active sessions.

Because LSASS contains authentication secrets, it is a primary target for attackers seeking to steal credentials or impersonate users.

Techniques such as credential dumping, Pass-the-Hash, and Pass-the-Ticket attacks exploit access to authentication data stored within LSASS.

Protecting LSASS through mechanisms such as Credential Guard, Protected Process Light, and strict privilege management is essential for maintaining identity security.

For defenders, monitoring interactions with LSASS, authentication logs, and credential-related behaviors provides critical visibility into identity-based attacks and lateral movement attempts within Windows environments.
